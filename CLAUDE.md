## What this project is

A self-directed lab for practising high-performance backend patterns in ASP.NET Core 9.
The goal is to build on the same product-catalogue domain, adding one scalability technique per iteration.

**Current iteration:** Nginx load balancer + 3 web-api replicas + Next.js 16 frontend

## Established patterns and conventions

**Backend (ASP.NET Core)**

| Convention | Detail |
|-----------|--------|
| Namespaces | File-scoped (`namespace Foo.Bar;`) |
| Async | `async/await` throughout — no `.Result` or `.Wait()` |
| Read queries | Always `.AsNoTracking()` |
| Seeding | Batches of 10,000 with `ChangeTracker.Clear()` after each `SaveChanges()` |
| Caching | Cache-Aside: check Redis → hit returns `source:"Cache"`, miss writes to Redis then returns `source:"Database"` |
| Pagination | `Skip((page-1)*pageSize).Take(pageSize)`, pageSize capped at 100 |
| Cache keys | `products_page_{page}_size_{pageSize}`, prefixed `ScaleLab:` via `InstanceName` |
| Cache TTL | 5-minute absolute expiry |
| Serialization | `System.Text.Json` with `JsonNamingPolicy.CamelCase` |

**Frontend (Next.js)**

| Convention | Detail |
|-----------|--------|
| Components | Always arrow functions — `const Foo = () => ...` then `export default Foo` |
| Client boundary | Push `'use client'` as deep as possible; keep data-fetching in Server Components |
| Streaming | Data-fetching components are async Server Components wrapped in `<Suspense>` |
| Error handling | `lib/api.ts` throws on non-ok responses; `error.tsx` catches them |
| Env var | `INTERNAL_API_URL` — set to `http://load-balancer` in Docker, fallback `http://localhost:5000` |

## Infrastructure

### Connection strings

| Environment | SQL Server | Redis |
|-------------|-----------|-------|
| `dotnet run` (local) | `Server=localhost,1433;...` | `localhost:6379` |
| Docker Compose | `Server=db,1433;...` | `redis:6379` |

Docker Compose overrides `appsettings.json` via `ConnectionStrings__DefaultConnection` and `ConnectionStrings__Redis` environment variables.

### Docker Compose ports

| Port | Service |
|------|---------|
| 3000 | Next.js frontend |
| 5000 | Nginx load balancer (API entry point) |
| 1433 | SQL Server (TDS — not HTTP) |
| 6379 | Redis |

Both SQL Server and Redis have health checks. The API uses `condition: service_healthy` for both so it never starts before they're ready.

### Startup sequence (Program.cs)

1. Register services (EF Core, Redis cache, controllers, Swagger)
2. Build and configure middleware pipeline
3. In a scoped block: run `db.Database.Migrate()`, then seed if `Products` table is empty

## EF Core migrations

Always run from the repo root:

```bash
dotnet restore
dotnet ef migrations add <MigrationName> \
  --project ScaleLab.Infrastructure \
  --startup-project ScaleLab.Presentation.Api
```

Migrations are applied automatically on startup via `db.Database.Migrate()`.
